Seção 2.1

2.1. Níveis de Teste

Os testes de software são divididos em etapas para isolar problemas e facilitar a correção. Os principais níveis de teste são: Unitário, Integrado, de Sistema e de Aceitação.

  • O Teste Unitário foca na menor parte testável do código, como uma única função, garantindo que ela faça o que deve isoladamente.
  • O Teste Integrado avalia como essas funções e módulos interagem entre si (por exemplo, o sistema salvando algo no banco de dados).
  • O Teste de Sistema valida o comportamento da aplicação inteira montada, de ponta a ponta.
  • Por fim, o Teste de Aceitação é focado no usuário final ou cliente, verificando se o software resolve o problema de negócio para o qual foi criado.
Seção 2.1 · Demonstração

Exemplo Prático: Cálculo de Desconto

Vamos imaginar o sistema da nossa SA e uma função simples de cálculo de desconto. Aqui está como testaríamos a menor unidade dela (Teste Unitário) em Python:

# A função de negócio (O que o sistema faz) def calcular_desconto(valor_total, cupom): if cupom == "SENA120": return valor_total * 0.80 # 20% de desconto return valor_total # O Teste Unitário (A validação) def test_calcular_desconto(): # Prepara os dados valor_compra = 100.0 # Executa a função resultado = calcular_desconto(valor_compra, "SENAI20") # Valida o resultado esperado assert resultado == 80.0, f'Erro: Esperado 80.0, mas retornou {resultado}' print("Teste Unitário passou com sucesso!") test_calcular_desconto()
Seção 2.1 · Análise

Análise do Teste Unitário

Explicação
A função calcular_desconto é a nossa unidade. O test_calcular_desconto é o nosso roteiro de teste automatizado. O comando assert verifica se o valor retornado pela função é igual ao que esperamos (80.0). Se for, o teste passa.
Debate
Se precisássemos testar a função calcular_desconto puxando a validade do cupom direto de um Banco de Dados, qual nível de teste estaríamos aplicando?

Seção 2.2

2.2. Testes Funcionais vs. Não Funcionais

É crucial entender as diferenças entre Testes Funcionais (o que o sistema faz) e Não Funcionais (como o sistema se comporta).

Testes Funcionais

Validam as regras de negócio:

  • o login funciona?
  • o botão de enviar manda o e-mail?
  • o cálculo do frete está correto?

Eles respondem se o sistema executa suas funções corretamente.

Testes Não Funcionais

Avaliam atributos de qualidade da aplicação:

  • O login suporta 1000 usuários simultâneos?
  • O sistema é seguro contra invasões?
  • Ele carrega em menos de 2 segundos?

Ele responde sobre o comportamento e a robustez da aplicação.

Seção 2.3

2.3. A Pirâmide de Testes

A Pirâmide de Testes serve como guia de eficiência. Ela ilustra como devemos distribuir nossos esforços de automação de testes para ter um projeto saudável, rápido e barato de manter.

  • A base da pirâmide é larga e representa os Testes Unitários: eles devem ser a grande maioria, pois são rápidos de executar e baratos de criar.
  • O meio da pirâmide representa os Testes de Integração: em menor quantidade que os unitários, pois são um pouco mais lentos e complexos.
  • O topo representa os Testes de UI (Interface/Sistema): devem ser poucos, apenas cobrindo as jornadas críticas, pois são muito lentos para rodar e difíceis de manter.
Seção 2.3 · Demonstração

Proporção Ideal de Testes

Se a sua equipe da SA tem 100 testes automatizados, a divisão ideal baseada na pirâmide seria algo como:

NívelQuantidadeDetalhe
Unitários70 testesValidando cada função matemática, validação de string, etc. Executam em milissegundos.
Integração20 testesValidando a comunicação da API com o Banco de Dados. Executam em alguns segundos.
Sistema/UI10 testesValidando o fluxo de ponta a ponta ("Abrir navegador, logar, comprar, sair"). Executam em vários minutos.
Atenção
Se a equipe inverter isso (Anti-padrão do Cone de Sorvete), o sistema demorará horas para ser testado e a entrega atrasará.
Debate
Por que os testes de Sistema (Interface gráfica) ficam no topo e em menor quantidade na pirâmide de testes?
Seção 2.4

2.4. Teste de Mesa e Ferramentas Interativas

Antes de automatizar ou mesmo codificar testes complexos, fazemos a prática de teste de mesa utilizando ferramentas interativas.

Como funciona
O teste de mesa é uma técnica manual onde simulamos a execução do código passo a passo, anotando no papel ou em planilhas o valor das variáveis a cada linha executada. Isso ajuda a encontrar erros de lógica visualmente e estimula o pensamento crítico antes do uso de compiladores.
Seção 2.4 · Prática

Simulação e Rastreamento

Demonstração Prática

Casos Reais do Mercado

Como Nubank e Netflix aplicam testes em seus sistemas.

Seção 4

Nubank e Netflix

Nubank e a Base da Pirâmide

As fintechs dependem criticamente de Testes Unitários. Imagine um erro de cálculo de juros aplicado a milhões de clientes! O Nubank possui milhares de testes unitários que rodam a cada modificação de código de um desenvolvedor. Se um único teste falhar, o código é bloqueado de ir para a produção.

Netflix e Testes Não Funcionais

A Netflix é pioneira em testes não funcionais de resiliência e performance. Eles têm um sistema chamado "Chaos Monkey", que desliga servidores de propósito no ambiente de produção para testar (em nível de sistema e integração) se a plataforma consegue se curar sozinha sem que o usuário final perceba o filme travando.

FAQ — Dúvidas Reais

Perguntas frequentes respondidas.

Seção 5

Perguntas Frequentes (1/2)

1. Posso testar a interface do usuário (UI) usando Testes Unitários?
Não é o ideal. Testes unitários focam no código lógico (regras de negócio). O teste de interface (ver se um botão é vermelho) pertence à camada de Testes de Sistema ou de UI (no topo da pirâmide).
2. Se eu fizer excelentes Testes de Sistema, preciso fazer Testes Unitários?
Sim! Testes de sistema são lentos e difíceis de rastrear. Se um teste de sistema falha, você sabe que o fluxo quebrou, mas é o teste unitário que aponta exatamente em qual linha de código a matemática falhou.
3. Um software pode falhar num teste Não Funcional, mas passar no Funcional?
Perfeitamente. O seu sistema pode fazer um PIX corretamente (Funcional Aprovado), mas demorar 5 minutos processando e consumir 100% do servidor, travando para os outros usuários (Não Funcional Reprovado).
Seção 5

Perguntas Frequentes (2/2)

4. Quem define o que o Teste de Aceitação vai validar?
Geralmente são os usuários finais, os Product Owners (PO) ou o cliente que encomendou o software, pois o objetivo é atestar se o software atende ao negócio.
5. Quando a gente começa a testar no projeto da SA?
Desde o primeiro dia! A mentalidade moderna de QA (Quality Assurance) adota o "Shift-Left", que significa trazer os testes o mais cedo possível, inclusive testando a ideia e os requisitos antes mesmo de codificar.
